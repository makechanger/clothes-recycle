# 衣物回收平台 - 分步实施计划（已更新版）

> 4 周 / 28 天 | 每步小而具体 | 每步包含验证方式 | 不包含代码
> 
> **本次更新已融入以下设计决策：**
> 1. 仅使用微信小程序测试号，登录和绑定采用**手动输入手机号**。
> 2. 三端（用户、回收员、机构）整合为**一个小程序，通过登录角色切换**展示不同 TabBar 和路由。
> 3. 积分发放降级策略：优先在“用户确认称重”时发放；第三周机构端完成后，升级为“机构扫码确认”后发放。
> 4. 回收员账号由**管理员在后台直接创建**，回收员登录后**上传资质**，管理员审核通过后方可接单。
> 5. 图片上传增加限制：前端压缩至 ≤1MB，后端限制 5MB。
> 6. 商城商品图片由后台上传，物流仅做“管理员填单号 + 用户看单号”。
> 7. 取消转运任务，回收员完成回收后**直接送达机构**进行扫码。

---

## 使用说明
- 每个步骤标注 **预计耗时**，帮助你判断进度是否正常。
- **验证** 栏描述该步骤完成后如何确认"做对了"。

- 优先级标记：🔴 P0 必须完成 | 🟡 P1 尽量完成 | 🟢 P2 有时间再做

## 第 1 周：基础框架 + 用户核心链路

> 周目标：用户可以微信登录、管理地址、提交回收预约、查看订单列表

### Day 1-2：环境搭建与项目初始化

#### Step 1.1 🔴 后端项目初始化（2h）
- 使用 Spring Initializr 创建 Spring Boot 3.2 项目。
- 勾选依赖：Spring Web、MySQL Driver、Lombok。
- 手动添加：MyBatis-Plus、Sa-Token、Knife4j、Hutool。
- 配置 `application.yml`：数据源、端口（8080）、Sa-Token 基础配置。
- **验证**：启动项目，浏览器访问 `http://localhost:8080` 返回 404；访问 `http://localhost:8080/doc.html` 能看到 Knife4j 文档。

#### Step 1.2 🔴 数据库创建与核心建表（3h）
- 安装 MySQL 8.0，创建数据库 `clothes_recycle`（字符集 utf8mb4）。
- 本周只建以下表：`user`（用户表）、`user_address`（用户地址表）、`recycle_order`（回收订单表）、`order_status_log`（订单状态流转记录表）、`community`（社区表）、`admin`（管理员表）。
- 为每张表添加主键自增、`created_at` 默认值、常用字段索引。
- 在 `community` 表中插入 3-5 条测试社区数据；在 `admin` 中插入一条默认管理员。
- **验证**：用数据库工具连接，确认所有表存在、字段类型正确。

#### Step 1.3 🔴 后端基础架构搭建（2h）
- 创建统一响应封装类 `Result<T>`。
- 创建全局异常处理器。
- 配置文件上传路径和静态资源映射（限制单文件最大 5MB）。
- **验证**：写一个测试接口返回 `Result.success("hello")`，Apifox 调用确认 JSON 格式正确。

#### Step 1.4 🔴 小程序项目初始化与角色路由骨架（3h）
- HBuilderX 创建 UniApp 项目（Vue 3 版本），安装 uView Plus 和 Pinia。
- 封装 `uni.request` 统一请求工具。
- **核心改动**：在 `pages.json` 中配置一套大路由，但利用 Pinia 存储当前登录用户的 `role`（USER / COLLECTOR / INSTITUTION）。
- 编写动态 TabBar 组件或在页面入口处根据 `role` 进行条件重定向。
- **验证**：HBuilderX 编译到微信开发者工具，能看到登录入口。

### Day 2-3：用户登录与个人信息

#### Step 1.5 🔴 后端 - 用户微信登录接口（3h）
- 依赖: Step 1.1, 1.2, 1.3
- 实现 `POST /api/user/login`：接收小程序传来的 `code`，调用微信 `jscode2session` 接口换取 `openid`。
- 根据 openid 查询用户表：存在则登录，不存在则自动注册。
- **核心改动**：由于是测试号，注册时手机号留空，提供单独的 `PUT /api/user/bindPhone` 接口供用户手动输入并绑定手机号（暂不接短信验证码，纯前端校验格式）。
- **验证**：Apifox 模拟登录请求，确认返回 token；调用绑定手机号接口，数据库中手机号成功更新。

#### Step 1.6 🔴 小程序 - 登录页与手动绑定手机（3h）
- 依赖: Step 1.4, 1.5
- 登录页：调用 `wx.login()` 获取 code -> 请求后端登录接口 -> 存储 token。
- **核心改动**：检测到用户未绑定手机号时，弹出手动输入手机号的表单，用户输入后调用后端绑定接口。
- **验证**：在微信开发者工具中点击登录，能成功获取 token并跳转首页；首次登录提示输入手机号，输入后能正常绑定。

### Day 3-4：地址管理

#### Step 1.7 🔴 后端 - 地址 CRUD 接口（2h）
- 实现地址管理四个接口（增、删、改、查），包含地址归属校验（只能操作自己的地址）。
- **验证**：Apifox 依次调用增删改查接口，确认数据正确写入；尝试操作别人的地址返回 403。

#### Step 1.8 🔴 小程序 - 地址管理页面（2h）
- 依赖: Step 1.4, 1.7
- 地址列表页与编辑页开发。省市区选择器使用 uView 的 `u-picker`。
- **验证**：新增一个地址 -> 列表中出现 -> 编辑修改 -> 确认更新 -> 删除 -> 确认消失。

### Day 4-5：回收预约下单

#### Step 1.9 🔴 后端 - 创建回收订单接口（3h）
- 实现 `POST /api/order/create`（参数含地址、预约时间段、衣物分类 JSON、预估重量、照片等）。
- 实现 `POST /api/common/upload`：通用文件上传接口，**限制文件大小不得超过 5MB**。
- **验证**：Apifox 调用上传接口，传入大于 5MB 的文件被拦截；调用创建订单接口，确认订单写入数据库且状态为"待接单"。

#### Step 1.10 🔴 小程序 - 回收预约页面（4h）
- 依赖: Step 1.8, 1.9
- 下单页面包含：选择地址、时间段、衣物分类多选、预估重量、上传衣物照片。
- **核心改动**：调用 `uni.chooseImage` 获取照片后，**在前端使用 canvas 进行压缩，确保单张图片大小 ≤1MB** 再调用上传接口。
- **验证**：填写完整信息提交，确认跳转到订单详情页；上传大图时，确认前端控制台打印了压缩日志且上传成功。

### Day 5-6：订单列表与详情

#### Step 1.11 🔴 后端 - 订单查询与取消（2h）
- 实现 `GET /api/order/list`（分页与状态筛选）和 `POST /api/order/{id}/cancel`（取消订单）。
- **验证**：创建 3 个不同状态的测试订单，调用列表接口确认分页和筛选正确。

#### Step 1.12 🔴 小程序 - 订单列表与详情页（3h）
- 订单列表页使用顶部 tab 切换不同状态（全部/待接单/进行中/已完成/已取消）。
- **验证**：列表页能正确展示订单，tab 切换筛选正确。

## 第 2 周：回收员端 + 订单全流程闭环

> 周目标：回收员端跑通，实现接单、上门称重、用户确认、直接送达机构

### Day 8：回收员数据准备

#### Step 2.1 🔴 后端 - 回收员账号管理接口（2h）
- **核心改动**：不提供回收员自主注册。实现管理员专用的 `POST /api/admin/collector` 接口，由管理员在后台直接创建回收员账号（指定初始手机号和密码）。
- **验证**：Apifox 调用该接口，数据库 `collector` 表成功插入一条回收员数据，状态默认为“待完善资质”。

#### Step 2.2 🔴 后端 - 回收员登录与资质上传（2h）
- 实现 `POST /api/collector/login`：手机号 + 密码登录（Sa-Token 独立 token）。
- 实现 `PUT /api/collector/profile/credentials`：回收员上传资质照片（身份证、健康证等 URL）。
- **验证**：用管理员创建的账号登录成功；上传资质照片后，回收员状态变更为“待审核”。

### Day 8-9：回收员接单

#### Step 2.3 🔴 后端 - 回收员订单接口（3h）
- 实现 `GET /api/collector/orders/pending`（查看待接单列表）和 `POST /api/collector/orders/{id}/accept`（接单）。
- **核心改动**：接单接口必须**校验回收员的 `status` 是否为“已认证”**。在本周自测时，可手动在数据库将测试回收员的状态改为“已认证”。
- **验证**：非“已认证”状态的回收员接单返回 403；状态改为“已认证”后，接单成功，订单状态变更为“已接单”。

#### Step 2.4 🔴 小程序 - 回收员登录与资质上传页（3h）
- 小程序登录页增加“回收员登录”入口。
- 回收员首次登录后，若检测到状态为“待完善资质”，强制跳转资质上传页面，要求手动输入姓名、上传身份证照片等。
- **验证**：回收员登录后，提示上传资质；上传完成后，页面提示“等待管理员审核”。

#### Step 2.5 🔴 小程序 - 回收员接单页面（3h）
- 资质通过（自测时手动改数据库）后，进入回收员端专属 TabBar 页面，查看待接单列表。
- **验证**：点击接单，订单进入“我的订单”列表。

### Day 9-10：称重录入与确认

#### Step 2.6 🔴 后端 - 称重与降级积分发放（3h）
- 实现 `POST /api/collector/orders/{id}/weigh`（录入实际重量和照片，状态 -> “待确认”）。
- 实现 `POST /api/order/{id}/confirmWeight`（用户端接口，状态 -> “已完成”）。
- **核心改动**：**在此步骤中，用户点击确认称重时，直接调用积分计算并发放给用户**（这是第一阶段的降级策略）。
- **验证**：用户确认称重后，订单状态变为“已完成”，且数据库中该用户的积分余额增加。

#### Step 2.7 🔴 小程序 - 回收员称重与用户确认页面（4h）
- 回收员端增加录入称重表单；用户端订单详情增加“确认称重”按钮。
- **验证**：回收员提交称重 -> 用户端看到数据 -> 用户点击确认 -> 流程结束。

### Day 10-11：溯源与直接送达

#### Step 2.8 🔴 后端 - 溯源码生成（2h）
- 实现 `POST /api/collector/orders/{id}/generateQR`：订单“已完成”后，生成唯一溯源码（格式：`TR` + 年月日 + 6位随机数），使用 ZXing 生成二维码图片，返回 URL。
- **验证**：调用接口生成二维码，手机扫码能读出溯源码内容。

#### Step 2.9 🔴 小程序 - 溯源码展示（1h）
- 回收员端订单完成后显示该二维码。
- **验证**：回收员在小程序上能看到生成的二维码。

### Day 11-12：第二周全流程联调

#### Step 2.10 🔴 订单全流程端到端测试（3h）
- 走通：用户下单 -> 资质已过的回收员接单 -> 回收员称重 -> 用户确认（积分发放）-> 回收员生成二维码（准备送往机构）。
- **验证**：整条链路无 Bug，数据库各状态流转日志完整。

## 第 3 周：积分规则 + 机构端 + 管理后台

> 周目标：积分发放策略升级（机构扫码后发放）、管理后台实现全方位管控

### Day 15：数据库补全

#### Step 3.1 🔴 数据库 - 补建积分与机构相关表（1.5h）
- 建表：`points_transaction`（流水）、`points_rule`（规则）、`institution`（机构表）。

### Day 15-16：机构端开发（升级积分策略）

#### Step 3.2 🔴 后端 - 机构登录与扫码接收接口（3h）
- 实现 `POST /api/institution/login`：机构账号登录（Sa-Token 独立 token）。
- 实现 `POST /api/institution/receive`：
  - 参数：`qr_code`。
  - **核心改动（积分策略升级）**：校验溯源码 -> 订单状态变更为“机构已接收” -> **此时再触发积分计算并给用户发放积分**。同时将 Step 2.6 处的积分发放逻辑注释或移除。
- **验证**：调用此接口，订单状态更新，用户积分增加。

#### Step 3.3 🔴 小程序 - 机构端页面（3h）
- 小程序登录页增加“机构登录”入口。
- 提供扫码接收页面，调用 `uni.scanCode` 获取二维码中的溯源码，传给后端。
- **验证**：机构登录 -> 扫描回收员出示的二维码 -> 提示接收成功。

### Day 16-17：管理后台搭建与审核

#### Step 3.4 🔴 管理后台项目初始化与登录（3h）
- clone vue-pure-admin 精简版模板。
- 对接后端 `POST /api/admin/login`。
- **验证**：后台能成功登录，进入骨架屏页面。

#### Step 3.5 🔴 管理后台 - 回收员账号创建与资质审核（3h）
- **核心改动**：实现回收员管理页面。管理员可以在此**新增回收员账号**，也可以看到状态为“待审核”的回收员，查看其资质照片，点击“通过”或“驳回”。
- **验证**：在后台新建回收员 -> 用该账号登录小程序上传资质 -> 后台刷新能看到 -> 点击通过 -> 小程序端回收员状态变为“已认证”。

### Day 17-20：管理后台核心管控

#### Step 3.6 🔴 管理后台 - 订单管理与积分规则（4h）
- 列表展示所有订单，多条件筛选。
- 提供积分规则配置表单（如：外套多少积分/kg）。
- **验证**：修改积分规则，新完成的订单按新规则计算积分。

### Day 21：第 3 周收尾

#### Step 3.7 🔴 升级后全链路集成自测（3h）
- **验证**：用户下单 -> 回收员接单并称重 -> 用户确认称重 -> 回收员带衣物直接去机构 -> **机构扫码接收** -> **用户积分正式到账**。
