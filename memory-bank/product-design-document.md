# 衣物回收平台 - 产品设计文档

> 毕业设计项目 | 开发周期：4 周 | 最后更新：2026-04-02

---

## 一、项目概述

### 1.1 项目背景

本项目为一个衣物回收平台，连接用户（捐赠衣物）、回收员（上门回收）、第三方机构（衣物接收处理）和平台管理方，实现衣物从预约下单、上门回收、称重溯源到积分激励的全链路闭环。

### 1.2 系统架构总览

```
┌─────────────┐  ┌─────────────┐  ┌─────────────┐
│  用户小程序   │  │ 回收员小程序  │  │ 第三方机构端  │
│  (微信端)    │  │  (微信端)    │  │  (微信端)    │
└──────┬──────┘  └──────┬──────┘  └──────┬──────┘
       │                │                │
       └────────┬───────┴────────┬───────┘
                │                │
         ┌──────▼──────┐  ┌─────▼──────┐
         │  后端 API    │  │ Web 管理后台 │
         │  (统一服务)   │  │  (管理员)   │
         └──────┬──────┘  └─────┬──────┘
                │               │
                └───────┬───────┘
                        │
                 ┌──────▼──────┐
                 │   数据库     │
                 │  (MySQL)    │
                 └─────────────┘
```

### 1.3 技术选型（建议）

| 层级 | 技术 | 说明 |
|:---|:---|:---|
| 用户端 / 回收员端 / 机构端 | 微信小程序（UniApp） | 一套代码多端复用 |
| Web 管理后台 | Vue 3 + Element Plus | 快速搭建管理界面 |
| 后端服务 | Spring Boot / Node.js | 统一 RESTful API |
| 数据库 | MySQL | 关系型数据，事务支持 |
| 对象存储 | 本地存储 / 云 OSS | 图片存储（称重照片等） |
| 二维码 | 后端生成 | 溯源码绑定订单 |

## 二、开发周期规划（4 周）

### 总体原则

- **MVP 优先**：先跑通核心链路，再补充辅助功能
- **纵向切片**：每周交付可演示的完整功能切片
- **砍掉非必要**：AI 图像识别、社区排行榜等降为"有时间再做"

### 2.1 功能优先级划分

| 优先级 | 说明 | 标记 |
|:---|:---|:---|
| P0 | 核心链路，必须完成，缺少则系统不可用 | 🔴 |
| P1 | 重要功能，完成后系统才完整 | 🟡 |
| P2 | 加分项，有时间再做 | 🟢 |

### 2.2 四周开发计划

#### 第 1 周：基础框架 + 用户核心链路

**目标：用户可以注册登录、提交回收预约订单**

- 🔴 项目脚手架搭建（前后端 + 数据库）
- 🔴 数据库设计与建表
- 🔴 用户微信登录 / 手机号绑定
- 🔴 用户个人信息管理（姓名、社区、地址）
- 🔴 回收预约下单（地址、时间段、衣物分类、估重、照片、备注）
- 🔴 订单列表与详情查看

#### 第 2 周：回收员端 + 订单全流程闭环

**目标：回收员可接单、上门称重、生成溯源码、完成订单**

- 🔴 回收员账号登录与认证信息展示
- 🔴 回收员接单 / 查看待处理订单
- 🔴 称重录入（手动输入 + 上传称重照片）
- 🔴 用户确认称重数据
- 🔴 溯源二维码生成与绑定
- 🔴 订单状态全流程流转（待接单 → 已接单 → 上门中 → 称重确认 → 已完成）
- 🟡 回收员扫码核验功能

#### 第 3 周：积分体系 + 第三方机构端 + 管理后台

**目标：积分发放闭环、机构可接收衣物、管理员可管控全局**

- 🔴 积分发放（订单完成后按重量自动计算并发放）
- 🔴 积分余额查询与收支明细
- 🟡 第三方机构登录与资质管理
- 🟡 机构接收任务查看与扫码确认收货
- 🔴 Web 管理后台：订单管理（列表、筛选、详情）
- 🔴 Web 管理后台：用户管理 / 回收员管理
- 🟡 Web 管理后台：积分规则配置

#### 第 4 周：补全 + 优化 + 答辩准备

**目标：补充重要功能、修复 Bug、准备演示**

- 🟡 积分兑换商城（商品列表、兑换下单、物流跟踪）
- 🟡 服务评价（用户对回收员评分评价）
- 🟡 异常订单申诉与处理
- 🟡 数据统计看板（订单量、回收总量、用户数）
- 🟢 机构选择功能（用户选择衣物去向机构）
- 🟢 环保报告 / 绿色账户
- 🟢 社区环保排行榜
- 🟢 AI 图像识别预检
- 🔴 Bug 修复与性能优化
- 🔴 答辩 PPT 与演示流程准备

## 三、功能模块详细设计

### 3.1 用户小程序端

#### 3.1.1 登录与个人信息

| 功能项 | 优先级 | 说明 |
|:---|:---|:---|
| 微信一键登录 | P0 | 调用微信授权获取 openid |
| 手机号绑定 | P0 | 微信手机号快捷验证 |
| 个人信息管理 | P0 | 姓名、所属社区（必填）、常用上门地址 |
| 地址管理 | P0 | 新增 / 编辑 / 删除常用地址 |
| 账号注销 | P1 | 清除未完成订单后可注销，脱敏处理个人数据 |

#### 3.1.2 回收预约（核心）

| 功能项 | 优先级 | 说明 |
|:---|:---|:---|
| 选择上门地址 | P0 | 从地址簿选择或新增 |
| 选择时间段 | P0 | 精确到 2 小时区间（如 9:00-11:00） |
| 衣物分类选择 | P0 | 多选：外套、裤子、鞋子、床品等 |
| 预估重量填写 | P0 | 用户自估，仅供参考 |
| 上传衣物照片 | P1 | 最多 9 张，辅助回收员判断 |
| 特殊备注 | P1 | 文字补充说明 |
| 回收规则公示 | P0 | 展示可回收范围、积分规则 |
| 机构选择 | P2 | 用户选择衣物去向的第三方机构 |
| AI 预检 | P2 | 拍照识别衣物材质、新旧程度 |

**预约业务规则：**

- 每日取消订单上限 3 次，超出当日无法发起新订单
- 回收员超出预约时间段 30 分钟未上门，用户可直接取消（不计入取消次数）
- 预估重量与实际重量偏差超过 ±30%，以现场核验为准

#### 3.1.3 订单管理

| 状态 | 说明 | 用户可执行操作 |
|:---|:---|:---|
| 待接单 | 订单已提交，等待回收员接单 | 取消订单 |
| 已接单 | 回收员已接单 | 取消订单、查看回收员信息 |
| 上门中 | 回收员正在上门途中 | 联系回收员 |
| 待确认 | 称重完成，等待用户确认 | 确认称重 / 申诉 |
| 已完成 | 订单完成，积分已发放 | 查看详情、评价、查看溯源码 |
| 已取消 | 订单已取消 | 查看详情 |
| 异常 | 存在争议，平台介入中 | 查看处理进度 |

#### 3.1.4 积分体系

| 功能项 | 优先级 | 说明 |
|:---|:---|:---|
| 积分余额显示 | P0 | 个人中心首页展示 |
| 积分收支明细 | P0 | 获取、兑换、扣除记录 |
| 积分商城 | P1 | 商品列表、积分兑换、兑换订单 |
| 兑换下单 | P1 | 选择商品 → 确认兑换 → 填写收货信息 → 生成兑换订单 |
| 兑换订单跟踪 | P1 | 物流进度查看，支持取消（未发货状态） |
| 绿色环保账户 | P2 | 累计回收总量、环保报告 |
| 社区排行榜 | P2 | 展示社区环保排行 |

**积分规则（默认配置，管理后台可调）：**

- 合格衣物：1 kg = 10 积分（示例值，可配置）
- 评价奖励：完成评价 +2 积分
- 积分兑换商品按件设定兑换门槛
- 积分扣除场景：兑换商品、异常订单仲裁扣除、违规扣除

#### 3.1.5 服务评价

| 功能项 | 优先级 | 说明 |
|:---|:---|:---|
| 订单评价 | P1 | 对上门准时度、服务态度、称重规范度打分 + 文字 |

| 异常申诉 | P1 | 称重不符、服务不规范、衣物丢失等问题提交 |

#### 3.1.6 个人中心

| 功能项 | 优先级 | 说明 |

## 五、核心业务流程

### 5.1 回收订单主流程

```
用户下单 → 回收员接单 → 回收员上门 → 现场称重
    → 双方确认重量 → 系统生成溯源二维码 → 回收员赋码绑定
    → 回收员提交完成 → 回收员转运至机构 → 机构扫码确认接收
    → 系统发放积分 → 订单闭环
```

### 5.2 订单状态机

```
                    ┌──────────┐
          ┌────────►│  已取消   │
          │         └──────────┘
          │
┌─────────┤  ┌──────────┐  ┌──────────┐  ┌──────────┐
│ 待接单   ├─►│  已接单   ├─►│  上门中   ├─►│  待确认   │
└─────────┘  └──────────┘  └──────────┘  └────┬─────┘
                                              │
                    ┌──────────┐  ┌────────┐  │  ┌──────────┐
                    │  异常     │◄─┤ 已完成  │◄─┘  │  已转运   │
                    └──────────┘  └───┬────┘      └────┬─────┘
                                      │                │
                                      ▼                ▼
                                 ┌──────────┐   ┌───────────┐
                                 │ 积分已发放 │   │ 机构已接收  │
                                 └──────────┘   └───────────┘
```

> 简化说明：称重确认后 → 已完成 → 回收员转运 → 机构接收 → 系统自动发放积分。

### 5.3 积分发放规则

1. 触发条件：第三方机构扫码确认接收完成
2. 计算方式：按积分规则表中对应衣物分类的 `points_per_kg × actual_weight`
3. 默认以回收员录入的确认重量执行；若机构确认重量与回收员录入不一致，以机构确认重量重新核算
4. 积分实时到账，用户可在订单详情、个人中心查看

### 5.4 取消订单规则

- 用户每日取消上限 3 次
- 超出预约时间段 30 分钟回收员未上门，用户取消不计入次数
- 积分兑换订单取消（未发货状态）：自动退还对应积分

## 七、非功能性需求

| 维度 | 要求 |
|:---|:---|
| 安全 | 接口鉴权（JWT）、敏感数据加密存储、SQL 注入防护 |
| 数据完整性 | 称重数据确认后锁定不可篡改、溯源码一般情况下不可修改解绑 |
| 并发 | 毕设场景无需高并发设计，单机部署即可 |
| 兼容性 | 微信小程序基础库 2.x+、管理后台支持 Chrome/Edge |
| 部署 | 单机部署即可，建议使用 Docker Compose 一键启动 |

## 九、风险与应对

| 风险 | 影响 | 应对策略 |
|:---|:---|:---|
| 开发周期不足 | 功能不完整 | 严格按 P0→P1→P2 顺序开发，P0 不完成不碰 P1 |
| 微信小程序审核 | 无法上线体验 | 使用开发者工具预览模式演示，答辩时不依赖线上版本 |
| 多端联调复杂 | 接口对不齐 | 先用 Mock 数据跑通前端，再对接真实后端 |
| 一人开发精力有限 | 质量下降 | 复用 UI 组件库（uView/Vant），减少重复工作 |
